````
它目前的含义是：

- `type: "Container"`

- Palette 里可以拖一个 Container 到画布

- 渲染时复用 Root 的结构，表现成一个带虚线边框的区域

Container
  layout: vbox | hbox | grid | stack | absolute

  padding
  align
  justify

```xml
<Control name="panel1" type="Container">
````

````
我的建议：`Container` 不应该只是单纯 layout，它应该是一个“可承载子控件的实体控件”。

更准确的关系是：

```text
Container = 控件实体 / 节点 / 子控件宿主
Layout    = Container 的布局策略
Frame     = 一种带可视边框/标题/语义的 Container
```

也就是说：

```xml
<Container name="toolbar" layout="hbox">
  <Button name="saveBtn" />
  <Button name="cancelBtn" />
</Container>
```

这里 `Container` 是实体，`hbox` 是它的布局方式。

**三者区别**
`Layout`：只描述怎么排子元素，比如 `vbox / hbox / grid / absolute / dock / flex`。它不是控件，不应该单独被脚本操作。

`Container`：是控件树里的实体。可以有 `name/id`，可以被选中、拖拽、绑定、隐藏、禁用、设置背景、边距、生命周期，也可以作为脚本对象存在，比如：

```js
panel1.visible = false
panel1.layout.gap = "8px"
```

`Frame / Panel / GroupBox`：是 Container 的一种可视化特例，比如有边框、标题、背景、滚动、折叠等。

**所以当前 OminiForge 里的 Container 应该理解为：**

它是一个“组合控件/容器控件”，不是单纯 layout。
现在它主要承担布局和承载子控件的能力，后续可以扩展成：

```text
Container
  chrome: none        // 纯布局容器，不显示边框
  layout: vbox

Frame
  chrome: frame       // 有边框/标题
  layout: vbox

Panel
  chrome: panel       // 有背景/内边距
  layout: absolute

ScrollView
  chrome: scroll
  layout: vbox
```

**我建议 DSL 层这样分：**

```text
BaseControl
  Button
  Label
  Container
    Frame
    Panel
    ScrollView
    TabView
```

`Container` 是基础组合能力，`Frame/Panel/ScrollView` 是带特定视觉或交互语义的容器控件。

所以结论是：`Container` 应该作为实体存在；`Layout` 是它的一个属性；`Frame` 是一种更具体、更有视觉语义的 Container。
````

````
- `sameWidth`：所有控件宽度等于 primary control 宽度
- `sameHeight`：所有控件高度等于 primary control 高度
- `distributeHorizontal`：按 x 排序，固定最左和最右，均分中间间距
````
