截至 `2026-04-26`，如果你们后续 `server` 要同时面对 `Web/QML client`、`CNC/PLC`、多协议接入和运维部署，我建议不要把所有职责都塞进一个进程。更合理的是分成四层：`southbound 设备接入`、`runtime 业务服务`、`northbound API`、`部署与观测`。

**核心判断**
- `HMI server` 最好做“监督控制 / 状态聚合 / 语义映射 / 指令编排”，不要做安全联锁或闭环运动控制。
- 面向浏览器/桌面客户端，主协议应该是 `HTTP + WebSocket`。
- 面向 PLC/CNC，主策略应该是“**优先标准协议，标准不够再用厂商 SDK/专有协议**”。
- 如果设备品牌很多，先上“工业网关/协议桥”通常比自己手写十几个驱动更划算。

**我建议的技术选型**
- `northbound Web 框架`：如果继续当前 `C++` 路线，首选 `Drogon`，因为它官方就支持异步、跨平台、`HTTP + WebSocket`；备选 `oat++`，优点是零依赖、DTO/OpenAPI/Swagger 体验更完整。  
- `southbound 标准协议`：优先 `OPC UA`。如果你们自己实现适配层，`open62541` 是很合适的基础库；它是 C 实现、可嵌到现有 C++ 服务里。  
- `CNC 语义标准`：如果你们更关心机床监控、OEE、程序状态、报警、产线汇聚，而不是直接控制轴运动，`MTConnect` 很值得上。它解决的是“不同机床数据语义统一”的问题。  
- `legacy/现场协议`：`Modbus RTU/TCP` 用 `libmodbus`；`EtherNet/IP`/部分 PLC 标签访问可用 `libplctag`；`FANUC` 系列可走 `FOCAS`。  
- `Siemens / Beckhoff`：如果控制器本身支持 `OPC UA Server`，优先直接接 `OPC UA`；`Beckhoff` 还可以走 `ADS`；老旧 `Siemens S7` 没有 OPC UA 时，再考虑 `Snap7` 这类社区库做兼容层。  
- `消息总线`：浏览器推送不要用 MQTT 替代 WebSocket。`MQTT/EMQX` 更适合站点级、边缘到云、系统解耦；`gRPC` 更适合内部服务之间高性能 RPC。  
- `数据库`：主库先用 `PostgreSQL`，存配置、权限、工单、报警、审计、程序元数据。高频时序遥测如果量很大，再上 `TimescaleDB` 或 `ClickHouse`。  
- `观测`：`OpenTelemetry C++` + `Prometheus` + `Grafana`，这套足够标准。  
- `反向代理/入口`：`Traefik` 很适合 `Docker/K8s` 场景，而且对 `WebSocket/WSS` 支持直接。  
- `低代码集成`：`Node-RED` 可以保留，但我建议只让它做外围编排、测试流、协议胶水，不要做主 runtime。

**三种可落地方案**
1. `最推荐，延续现有代码`
   - `C++ runtime + Drogon + WebSocket`
   - `open62541` 做 OPC UA 适配
   - `libmodbus / libplctag / FOCAS` 做 legacy 或厂商协议
   - `PostgreSQL + Prometheus/Grafana`
   - `Docker Compose + Traefik`
   - 这个方案改造成本最低，最贴近你们现在仓库。

2. `多厂商、快交付`
   - 设备侧不上自研十几个驱动，先用 `EMQX Neuron` 这类工业网关收敛 `OPC UA / Modbus / S7 / EtherNet/IP`
   - 你们的 `HMI server` 只做语义模型、权限、命令编排、HTTP/WS
   - 好处是设备接入风险最低，缺点是引入外部平台组件。

3. `平台化、后期大规模扩展`
   - `runtime service` 继续用 `C++` 或改 `Rust`
   - 浏览器侧 `HTTP + WS`
   - 服务间 `gRPC`
   - 站点/云流转 `MQTT` 或 `NATS`

- 单厂区、单边缘节点、服务不多时，先用 `Docker Compose`，不要一开始就上 `Kubernetes`。

**方案对比**

|---|---|---|---|---|---|---|

- `Web API 框架`
  - 首选 `Drogon`。官方就支持异步、HTTP、WebSocket，继续走 `C++` 成本最低。
  - `oat++` 可以作为备选，如果你们更看重 DTO/OpenAPI 这类接口治理。
- `客户端通信`
  - 对 `Web/QML`：`HTTP + WebSocket`
  - `HTTP` 负责 `bootstrap/health/commands/property/resource`
  - `WebSocket` 负责 `state delta/alarms/events/async result`
  - 不建议继续把高频状态同步主要压在轮询上。
- `设备接入优先级`
  - `1. OPC UA`：能用就优先。它最适合 PLC/CNC/HMI 之间做标准化、权限和信息模型。
  - `2. MTConnect`：如果重点是机床监控、程序状态、报警、产线数据汇聚，而不是细颗粒控制，它很合适。
  - `3. 厂商 SDK / 专有协议`：比如 `FANUC FOCAS`、部分 Beckhoff/Siemens 特有链路，只在标准协议不够时补上。

- `数据存储`
