我建议分三层落：

1. **每个已实现/部分支持代码的详细页**
   比如 `G0/G1/G2/G3/G4/G17/G18/G19/G40/G41/G42/M6/M17/M98`。每页包含：
   - 支持状态
   - 语法形式
   - 参数/word
   - lowering 到什么 AIL/Message
   - ExecutionSession 行为
   - 正常示例
   - 错误示例
   - corner cases
   - 对应测试/fixture

2. **完整场景 cookbook**
   不按单个代码，而按任务：
   - 线性加工段
   - 平面切换后圆弧
   - runtime 变量读取并阻塞恢复
   - 延迟换刀 `T + M6`
   - `IF/ELSE/ENDIF` 分支
   - 子程序调用和返回
   - rejected line 后编辑恢复

3. **corner case / acceptance matrix**
   这部分很有价值，尤其适合后续测试驱动：
   - `G1 G2 X10` 多运动码冲突
   - `G4 F3 S30`
   - `G4 F3 X10`
   - `G2` 在 `G18` 下用了 `J`
   - `AP90` 缺 `=`
   - `/10` 非法 skip level
   - `G1 N100 X1` 行号位置错误
   - `M98 P1000` 未开 ISO mode
   - `T=DRILL` 在 `tool_management=false`
   - `$P_UIFR[1,X,TR]` 多 selector 不支持
   - `RET` 空调用栈
   - unresolved `GOTO`

关键是：**详细说明只给已实现/部分支持的代码写实页；规划项只保留索引和状态，不展开成“可用手册”。** 否则文档会比实现先跑远，后面很难维护。
