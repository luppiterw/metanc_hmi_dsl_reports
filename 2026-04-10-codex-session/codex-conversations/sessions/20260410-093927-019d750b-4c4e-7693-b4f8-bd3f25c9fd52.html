17. Codex / final_answer

2026-04-10 09:49:45 +0800
我按当前代码和文档核了一遍。结论是：现在已经不是“空架子”，而是一个可运行的刀具库存、寿命、搜索、装卸和持久化基线；但离完整 SINUMERIK 风格刀具管理还差机器侧状态、报警、换刀流程、PLC/NC/HMI 集成这些核心层。

**已做**
当前已经实现并有测试覆盖的部分：

- 刀具主数据：`id`、`tool_number`、`name`、`type`、多刃口、当前位置等，见 [model.hpp](/home/iaar/workspace/codex-wp/metanc_tool_man/include/toolman/model.hpp):10。
- 刃口数据和寿命计数：每个 edge 有 `warning_limit`、`limit`、`used`，能判断预警和到限。
- 刀库和槽位：固定容量刀库，槽位 1-based 连续编号。
- 显式装刀/卸刀：按 `magazine_id + slot_index` 装卸，拒绝重复装载、占用槽位等非法状态。
- 用量记录：可以对某个刀具刃口累计使用量。
- 状态导入/导出：`GET /state`、`POST /state/import`，并校验刀具位置和刀库槽位一致性。
- 自动持久化：JSON 文件后端和 SQLite 后端都已经有了；SQLite 现在也能通过 CMake 自动走系统库或 bundled 子模块。
- HTTP API：刀具、刀库、装卸、使用量、搜索、状态导入导出、健康检查都有路由，见 [http-api.md](/home/iaar/workspace/codex-wp/metanc_tool_man/docs/http-api.md):24。
- 替换刀具基础能力：
  - `group_id`
  - `replacement_priority`
  - `wear_group_id`
  - `search_enabled`
  - `replacement_tool_id` 显式替换链
- 搜索基础能力：
  - `priority_first`
  - `healthy_first`
  - `wear_balanced`
  - `inventory_scope`: `any` / `loaded_only` / `unloaded_only`
  - `health_scope`: `any_usable` / `healthy_only`
  - 可返回已装载且可用的刀具
  - 可报告 `requested_tool_id` 和 `replacement_path`
- 空槽搜索和辅助装刀：
  - `lowest_slot`
  - `highest_slot`
  - `nearest_slot`
- 自动化测试覆盖比较实：管理器规则、HTTP 路由、错误合同、文件持久化、SQLite 持久化、搜索策略、替换链都有测试入口。

**部分完成**
这些已经有雏形，但还不是完整刀具管理：

- 刀具寿命监控：现在只是手动 `record usage`，没有机器侧自动累计、加工事件、换刀触发、寿命到限联动。
- 刀具搜索策略：现在是仓库自定义的三种策略，不是可配置的完整 Siemens 搜索定义系统。
- 替换刀具/替换组：有组、优先级、替换链、磨损组，但还没有完整备用刀策略、适配条件、锁定状态、禁用原因、刀具等级等。
- 空位置搜索：有最低、最高、最近槽位，但没有完整的刀库区、装载点、换刀点、当前位置、距离/运动约束。
- 持久化：现在是 snapshot 级别，够用但不是增量事件、事务日志或多客户端数据库模型。
- HTTP 合同：是项目自己的 API，不是 Siemens NC/PLC/Operate 原生接口。

这些在 [manual-implementation-status.md](/home/iaar/workspace/codex-wp/metanc_tool_man/docs/manual-implementation-status.md):20 里也被标成 `Partial`。

**未做**
刀具管理真正接近机床时，下面这些还没开始：

- 主轴/当前刀模型：没有 spindle、active tool、prepared tool、next tool。
- 换刀流程：没有换刀请求、预选刀、执行中、完成、失败、回滚等状态机。
- 刀库运动：没有刀库定位、换刀点、搬运、机械手、重定位、交换槽。
- PLC 接口：没有信号、DB、握手、互锁、M 功能触发。
- NC 接口：没有 NC 指令、系统变量、OPI/通道级交互。
- 报警系统：没有报警码、报警状态、确认、复位、事件历史。
- 机床数据/调试配置：没有 commissioning 参数、刀库类型、容量配置、搜索规则配置。
- 权限/保护级：没有用户、角色、写操作保护。
- HMI/Operate：没有界面层或 HMI view model。
- RFID/刀具识别：没有 code carrier、外部识别、扫码/RFID 装刀校验。
- 刀具几何和补偿还很薄：现在只有 `length_offset`、`radius` 这种占位级字段，没有完整 D 号、补偿号、刀沿方向、姐妹刀数据等。

这些限制在 [architecture.md](/home/iaar/workspace/codex-wp/metanc_tool_man/docs/architecture.md):73 也有记录。

**我建议下一步**
不要直接跳 PLC/NC。现在最值得做的是把“库存智能”和“机器状态”之间的中间层补起来。

1. 先收尾搜索/替换规则  
   做一个持久化的 `SearchProfile` / `ReplacementPolicy` 模型，支持命名策略、库存范围、健康范围、优先级规则、允许/禁止已装刀、允许/禁止预警刀、槽位搜索规则。这样现在硬编码的 `priority_first` 等策略能变成配置。

2. 加刀具状态和锁定原因  
   给刀具增加 `enabled/locked/blocked/reason` 这类状态。当前只有 `search_enabled`，太粗。实际需要区分：人工禁用、寿命到限、测量未完成、位置未知、装载中、PLC 锁定等。

3. 做报警和事件  
   先从寿命报警做起：使用量达到 warning/limit 时生成 retained alarm/event。加 `GET /alarms`、确认报警、清除已恢复报警。这个会让“寿命监控”从 Partial 往 Implemented 走。

4. 建主轴和换刀请求模型  
   增加 `MachineState`：当前主轴刀、候选刀、换刀请求、换刀状态。先不接 PLC，只在服务内部建状态机，例如 `requested -> prepared -> loaded -> completed/failed`。

5. 做刀库位置/搬运动作的抽象  
   在有主轴状态后，再做 magazine positioning：换刀点、装载点、当前位置、目标槽位、移动命令状态。否则现在的 load/unload 只是库存突变，不像真实机床。

6. 最后再接 PLC/NC/HMI/RFID  
   等 retained state、报警、换刀状态稳定后，再定义 adapter。否则接口会频繁返工。

如果只选一个最近的开发目标，我建议做第 1 和第 2：**把搜索/替换策略配置化，同时补刀具状态/锁定原因**。这是现有代码自然延伸，测试也好写，并且会直接提升刀具管理核心能力。
