这些是日志系统从“能用”走向“可维护/可运维”的后续能力：

- **offline client log buffer**
  Web/QML 客户端断网、server 没启动、WS/HTTP 失败时，先把客户端日志临时存在本地。等 server 恢复后再上传。否则故障发生时最关键的 client 端日志可能丢掉。

- **batch upload: `POST /api/runtime/logs/client/batch`**
  现在 client 是单条上传。batch upload 是一次上传多条日志，适合 offline buffer 恢复后批量补传，也能减少 HTTP 请求数量。通常还要配 `client_event_id` 去重，避免断线重传导致重复日志。

- **JSONL export endpoint**
  提供类似：
  ```text
  GET /api/runtime/logs/export?format=jsonl
  ```
  把日志导出成一行一个 JSON 的文件格式，方便排障、归档、发给开发分析，或者导入别的日志系统。

- **retention 策略和 retention runner**
  retention 就是“日志保留多久”。比如 client debug 只保留 1-3 天，server error 保留 30 天，audit 保留 90 天或更久。runner 是执行清理的任务，可以手动触发或未来定时跑。

- **server-side clear policy**
  UI 上的“清空日志”不能随便删所有日志，尤其不能删 audit。server-side clear policy 就是规定哪些日志允许清，比如只清 client debug/info，不能清 command audit、setting change、tool change 这类关键记录。

- **audit policy enforcement**
  audit 是审计日志，记录重要操作：启动程序、修改设置、修改刀具、修改参数等。enforcement 是确保这些日志不能被普通清理接口删除，字段必须完整，保留时间按策略执行，后续可追责。

- **时间范围查询和正式 cursor pagination**
  现在查询主要靠 `after_id + limit` 和一些字段过滤。后续需要支持：
  ```text
  since=2026-04-30T10:00:00Z
  until=2026-04-30T11:00:00Z
  cursor=...
  ```
  这样 UI 可以按时间查、翻页、加载更多，而不是一次拉很多或者只看最新窗口。
