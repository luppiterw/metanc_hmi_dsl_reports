2026-04-28 11:20:47 +0800
现在 Web 端使用方式是：只要进入 `strict` server 模式，它会自动从 HTTP bootstrap 切到 WebSocket 订阅。

最简单启动方式：

```bash
cd /home/iaar/workspace/ccmix-wp/metanc_hmi_dsl
./generated/distribution/run_split_web_native.sh 8010 8000
```

然后打开：

```text
http://127.0.0.1:8000/?server=http://127.0.0.1:8010/api/runtime
```

这里实际发生的是：

1. Web 先请求 `GET /api/runtime/bootstrap`
2. bootstrap 成功后自动连接：
   ```text
   ws://127.0.0.1:8010/api/runtime/ws
   ```
3. 连接打开后 Web 端发送：
   ```json
   {
     "type": "runtime.subscribe",
     "domains": ["state", "commands"],
     "since_revision": 0
   }
   ```
4. server 会先回：
   ```text
   runtime.subscription.ready
   runtime.snapshot
   ```
5. 之后 REST 命令、属性/resource 变更会通过 WS 推：
   ```text
   runtime.state.changed
   runtime.command.requested
   runtime.command.completed
   ```

如果你已经单独启动了 server，只启动 Web 客户端可以这样：

```bash
./generated/distribution/run_client_web.sh --restart 8000 http://127.0.0.1:8010/api/runtime
```

关键点是第二个参数 `http://127.0.0.1:8010/api/runtime`。脚本会生成临时 `config.js`，把 Web 设置成：

```js
window.HMI_RUNTIME_CONFIG = {
  serverMode: "strict",
  serverBaseUrl: "http://127.0.0.1:8010/api/runtime",
};
```

代码入口在 [runtime_shell.py](/home/iaar/workspace/ccmix-wp/metanc_hmi_dsl/client/web_client/runtime_shell.py:2126)：`startSubscription()` 会创建 WebSocket；[runtime_shell.py](/home/iaar/workspace/ccmix-wp/metanc_hmi_dsl/client/web_client/runtime_shell.py:2199) 发送 `runtime.subscribe`；[runtime_shell.py](/home/iaar/workspace/ccmix-wp/metanc_hmi_dsl/client/web_client/runtime_shell.py:2299) 负责把 HTTP base URL 转成 WS URL。

如果 WS 断了但已有 snapshot，Web 会自动降级成 HTTP polling，并显示：

```text
HMI Server WebSocket unavailable; using HTTP polling
```

对，但更精确地说是三段：

1. **Client 先知道要看哪些数据**
   Web 端是根据生成出来的 `runtime_plan`/binding path 渲染界面，比如 `axis.x.pos_machine`、`prog.line_no`、`runtime_state.execution_state` 这些路径。

2. **Client 连接 server 后订阅状态**
   启动时先 `GET /api/runtime/bootstrap` 拿一次完整初始状态；然后通过 WebSocket 发送：
   ```js
   {
     type: "runtime.subscribe",
     domains: ["state", "commands"],
     since_revision: ...
   }
   ```
   也就是订阅 server 的状态和命令事件。

3. **Server 后续 pub 状态变化**
   Server 每次命令执行、模拟 tick 推进、状态变化后，会通过 WS 推送 `runtime.snapshot` / `runtime.state.changed` / `runtime.command.completed` 这类消息，里面带 `state`。Client 收到后把 `properties/resources/streams/local_state` 合并进界面状态。

所以可以说：**现在 client 是 sub server 的 runtime state，server pub 状态快照/变化过来。**

但不是完全只靠 WS：
- 初始数据：HTTP `/bootstrap`
- 点击按钮命令：HTTP POST `/commands`
- 后续自动变化：WS subscription 推送
- WS 不可用时 strict 模式还有 polling 兜底 `/state`

当前 pub 出来的数据源是 server 里的 `SimulatorAdapter`，不是浏览器本地 mock，也还不是实际 CNC。
