```bash
cd /home/iaar/workspace/ccmix-wp/metanc_hmi_dsl
./generated/distribution/run_split_web_native.sh 8010 8000
```

然后打开：

```text
http://127.0.0.1:8000/?server=http://127.0.0.1:8010/api/runtime
```

这里实际发生的是：

1. Web 先请求 `GET /api/runtime/bootstrap`
2. bootstrap 成功后自动连接：
   ```text
   ws://127.0.0.1:8010/api/runtime/ws
   ```
3. 连接打开后 Web 端发送：
   ```json
   {
     "type": "runtime.subscribe",
     "domains": ["state", "commands"],
     "since_revision": 0
   }
   ```
4. server 会先回：
   ```text
   runtime.subscription.ready
   runtime.snapshot
   ```
5. 之后 REST 命令、属性/resource 变更会通过 WS 推：
   ```text
   runtime.state.changed
   runtime.command.requested
   runtime.command.completed
   ```

如果你已经单独启动了 server，只启动 Web 客户端可以这样：

```bash
./generated/distribution/run_client_web.sh --restart 8000 http://127.0.0.1:8010/api/runtime
```

```js
window.HMI_RUNTIME_CONFIG = {
  serverMode: "strict",
  serverBaseUrl: "http://127.0.0.1:8010/api/runtime",
};
```

代码入口在 [runtime_shell.py](/home/iaar/workspace/ccmix-wp/metanc_hmi_dsl/client/web_client/runtime_shell.py:2126)：`startSubscription()` 会创建 WebSocket；[runtime_shell.py](/home/iaar/workspace/ccmix-wp/metanc_hmi_dsl/client/web_client/runtime_shell.py:2199) 发送 `runtime.subscribe`；[runtime_shell.py](/home/iaar/workspace/ccmix-wp/metanc_hmi_dsl/client/web_client/runtime_shell.py:2299) 负责把 HTTP base URL 转成 WS URL。

```text
HMI Server WebSocket unavailable; using HTTP polling
```

原因很直接：现在 Web 是“生成型静态客户端”，不是一个独立前端应用。Python 生成器负责把 DSL/IR 变成 `generated/web`，最终还要进入 `generated/distribution`。这种场景下，esbuild 更贴合。

**npm + esbuild 更适合当前阶段：**

- 产物简单：输入 `src/main.ts`，输出一个 `app.bundle.js`。
- 很容易保持静态分发：`index.html + bundle + model.generated.json`。
- 不强依赖 dev server，和现在 `run_split_web_native.sh 8010 8000` 这种方式容易兼容。
- 第三方库接入也够用，比如 CodeMirror 6：
  ```bash
  npm install @codemirror/state @codemirror/view @codemirror/commands @codemirror/language
  ```
- Python 生成器不需要理解 npm，只要生成数据文件；JS 构建负责打包组件和第三方库。
- CI/快照测试更稳定，bundle 产物可控。

**Vite 更适合后一个阶段：**

如果 Web 客户端以后变成一个独立产品级前端，比如需要：

- 长期热更新调试
- 多页面/多入口
- 复杂 CSS/资源处理
- 插件生态
- 更完整的 dev server 体验
- 甚至未来引入 React/Vue/Svelte

那 Vite 会更舒服。但它会把当前生成链路变成“生成器 + 前端应用工程 + Vite 构建”的模式，复杂度会上一个台阶。

我的建议是：

```text
当前：npm + esbuild
以后 Web 客户端明显膨胀后：再迁移到 Vite
```

具体落地我会这样设计：

```text
client/web_client/
  package.json
  package-lock.json
  esbuild.config.mjs
  src/
    main.ts
    runtime/
    widgets/
    editors/
      program-editor.ts
      codemirror-program-editor.ts
      textarea-program-editor.ts
    styles/
      app.css
```

生成后：

```text
generated/web/
  index.html
  assets/app.bundle.js
  assets/app.css
  model.generated.json
  runtime_seed.generated.json
```

也就是说：**用 npm 管第三方包，用 esbuild 打包，用 Python 继续管 DSL 生成。** 这是当前风险最低、收益最大的方案。
